# 产品综合工作流

本技能是产品工作的总控入口，不直接替代所有专项技能，而是负责：

1. 判断当前任务类型
2. 读取正确的文档与输入
3. 选择正确的流程阶段
4. 在需要时路由到专项技能

## 工作目标

本技能的目标不是直接产出所有结果，而是先判断清楚以下四件事：

1. 当前任务属于哪一类产品工作
2. 当前任务应该先走哪一个阶段
3. 当前最应该读取哪份文档
4. 当前是否必须切换到专项技能

只有先把这四件事判断清楚，后续产物才不会跑偏。

## 默认触发场景

以下任务默认先走本技能：

- 需求收口与范围澄清
- 原型规划与页面结构梳理
- 设计规范 / 原型规范 / DESIGN.md 协同
- PRD 前的输入整理
- 评审前的结构检查
- 规范沉淀、回写、流程判断
- 用户只说“这个产品任务怎么做”“下一步该走什么流程”

以下任务由本技能判断后再路由，不直接在本技能内完成最终产出：

- 写 PRD / 生成 PRD / 按目录出文档 -> `zz-pm-prd`
- 原型 / 页面复刻 / 截图做页面 / URL 做原型 -> `zz-pm-prototype-workflow`
- 评审 / review / 查漏补缺 / 过会前检查 -> `zz-pm-review-board`
- 埋点 / tracking / 事件字段设计 / 指标口径 / QA 验收 -> `zz-pm-tracking-spec`
- A/B 测试 / 实验设计 / 灰度方案 / 样本量 / 判定规则 -> `zz-pm-experiment`
- 复盘 / postmortem / RCA / 上线回顾 / 行动项沉淀 -> `zz-pm-postmortem`
- 优先级 / RICE / ICE / Kano / 取舍 -> `zz-pm-prioritization`
- roadmap / 版本规划 / 里程碑 / 依赖梳理 -> `zz-pm-roadmap`
- 数据分析 / 漏斗 / 留存 / 指标异常 -> `zz-pm-analytics`
- 竞品分析 / 对标 / 差异化 -> `zz-pm-competitor`
- 问卷设计 / survey / 满意度调研 -> `zz-pm-survey`
- 本地 Word 文档输出 -> `docx` + `zz-pm-prd`
- UI / 原型 / 页面还原 -> 先判断是否需要进入原型流程，再选择相关规范或技能

## 核心路由规则

### 0. 任务分类矩阵

收到产品任务后，优先归入下列 15 类之一：

| 类型 | 典型输入 | 主输出 | 默认动作 |
|---|---|---|---|
| 需求收口 | “这个需求怎么做”“帮我梳理范围”“有哪些待确认项” | 已知信息、待确认项、阶段判断 | 先收口，不直接写 PRD |
| 原型子系统 | “帮我做原型”“根据截图做页面”“根据 URL 复刻” | 原型路径判断与交付形式 | 立即切到 `zz-pm-prototype-workflow` |
| 原型规划 | “这个页面怎么画”“先出结构”“原型该到什么粒度” | 页面模板、结构草稿、状态清单 | 先走原型与结构 |
| 设计协同 | “按现有设计规范做”“这个页面风格怎么统一” | 设计输入优先级、规范清单 | 先读 `DESIGN.md` 与设计规范 |
| PRD 产出 | “写 PRD”“生成正式版”“按目录出文档” | 初评版或正式版 PRD | 立即切到 `zz-pm-prd` |
| 评审准备 | “帮我过一遍”“查漏补缺”“这个方案能不能评审” | 风险点、补充项、检查清单 | 先列问题，不直接重写 |
| 埋点设计 | “这个功能怎么埋点”“帮我写 tracking spec” | 事件清单、字段、口径、验收项 | 立即切到 `zz-pm-tracking-spec` |
| 实验设计 | “帮我做 A/B 测试”“这次怎么灰度” | 假设、分组、指标、样本量、止损规则 | 立即切到 `zz-pm-experiment` |
| 复盘总结 | “帮我写复盘”“这次上线怎么总结” | 偏差归因、行动项、经验沉淀 | 立即切到 `zz-pm-postmortem` |
| 优先级决策 | “帮我排优先级”“这些需求先做哪个” | 排序、取舍和版本建议 | 立即切到 `zz-pm-prioritization` |
| 路线图规划 | “帮我做 roadmap”“这季度怎么排” | 里程碑、依赖、缓冲方案 | 立即切到 `zz-pm-roadmap` |
| 数据分析 | “帮我分析数据”“这个指标为什么跌了” | 关键发现、假设、动作建议 | 立即切到 `zz-pm-analytics` |
| 竞品拆解 | “帮我做竞品分析”“和竞品比一下” | 对比矩阵、差异化建议 | 立即切到 `zz-pm-competitor` |
| 问卷调研 | “帮我设计问卷”“做个用户调研” | 研究假设、题目、发放建议 | 立即切到 `zz-pm-survey` |
| 规范沉淀 | “把这轮结论回写”“更新规范” | 应回写的文档与更新范围 | 先判断更新对象 |

如果一个任务同时命中多类，优先级如下：

1. `PRD 产出`
2. `需求收口`
3. `原型子系统`
4. `原型规划`
5. `设计协同`
6. `评审准备`
7. `埋点设计`
8. `实验设计`
9. `复盘总结`
10. `优先级决策`
11. `路线图规划`
12. `数据分析`
13. `竞品拆解`
14. `问卷调研`
15. `规范沉淀`

### 1. 先判断任务类型

优先按以下顺序判断：

1. 当前是不是 `PRD / 正式文档交付`
2. 当前是不是 `需求收口 / 页面范围澄清 / 待确认项整理`
3. 当前是不是 `截图 / URL / .pen / 页面复刻类原型任务`
4. 当前是不是 `原型结构 / 页面模板 / 状态覆盖`
5. 当前是不是 `设计规范 / DESIGN.md / 可视化规范资产协同`
6. 当前是不是 `评审 / review / 过会前检查`
7. 当前是不是 `埋点 / tracking / 字段和指标口径设计`
8. 当前是不是 `A/B 测试 / 实验设计 / 灰度验证 / 样本量估算`
9. 当前是不是 `复盘 / postmortem / RCA / 上线总结`
10. 当前是不是 `优先级 / RICE / ICE / Kano / 取舍`
11. 当前是不是 `roadmap / 版本规划 / 里程碑 / 依赖编排`
12. 当前是不是 `数据分析 / 漏斗 / 留存 / 指标异常`
13. 当前是不是 `竞品分析 / 对标 / 差异化`
14. 当前是不是 `问卷 / survey / 满意度 / 调研设计`
15. 当前是不是 `回写 / 规范沉淀`

### 2. 按类型读取文档

若项目内存在，默认读取顺序如下：

1. `docs/*SOP*.md`
2. `DESIGN.md`
3. `docs/*设计规范*.md`
4. `docs/*原型规范*.md`
5. `scope.md / rules.md / glossary.md / examples/ / screenshots/ / prototype/`

规则：

- 需要 AI 生成界面或原型时，优先读 `DESIGN.md`
- 需要团队协作口径时，读长版 `设计规范.md`
- 需要真实视觉、组件排列、画板示例时，查看 `.pen`
- 需要正式 PRD 输出时，切换到 `zz-pm-prd`

### 3. 路由到子技能

当任务命中以下情况时，直接切换：

- `写 PRD / 生成 PRD / 初评版 PRD / 正式版 PRD / 按目录出文档`
  - 路由到 `zz-pm-prd`
- `原型 / 截图复刻 / URL 复刻 / .pen / 页面复刻`
  - 路由到 `zz-pm-prototype-workflow`
- `评审 / review / 查漏补缺 / 过会 / 评审前检查`
  - 路由到 `zz-pm-review-board`
- `埋点 / tracking / 事件设计 / 字段设计 / 指标口径 / QA 验收`
  - 路由到 `zz-pm-tracking-spec`
- `A/B 测试 / 实验设计 / 灰度方案 / 样本量 / 判定标准`
  - 路由到 `zz-pm-experiment`
- `复盘 / postmortem / RCA / 上线复盘 / 项目复盘`
  - 路由到 `zz-pm-postmortem`
- `优先级 / RICE / ICE / Kano / 取舍 / 哪些先做`
  - 路由到 `zz-pm-prioritization`
- `roadmap / 版本规划 / 里程碑 / 关键路径 / 依赖梳理`
  - 路由到 `zz-pm-roadmap`
- `数据分析 / 漏斗 / 留存 / 分群 / 指标异常`
  - 路由到 `zz-pm-analytics`
- `竞品分析 / 对标 / 差异化 / competitor analysis`
  - 路由到 `zz-pm-competitor`
- `问卷设计 / survey / 满意度调查 / NPS`
  - 路由到 `zz-pm-survey`
- `本地 Word 文档交付`
  - 路由到 `docx`，并以 `zz-pm-prd` 作为内容规则来源

### 4. 必须切换子技能的硬边界

以下情况不应继续停留在本技能内：

- 用户明确要求 `PRD`、`正式版文档`、`初评版文档`
- 用户明确要求 `原型`、`截图做页面`、`URL 复刻`、`.pen`
- 用户要求按目录、按模板、按截图或按原型输出正式文档
- 用户要求本地 `docx` 交付
- 用户明确要求 `评审`、`review`、`过会前检查`、`查漏补缺`
- 用户明确要求 `埋点`、`tracking`、`事件设计`、`指标口径`、`QA 验收`
- 用户明确要求 `A/B 测试`、`实验设计`、`灰度方案`、`样本量`、`判定标准`
- 用户明确要求 `复盘`、`postmortem`、`RCA`、`上线总结`、`行动项沉淀`
- 用户明确要求 `优先级`、`RICE`、`ICE`、`Kano`、`取舍`
- 用户明确要求 `roadmap`、`版本规划`、`里程碑`、`关键路径`
- 用户明确要求 `数据分析`、`漏斗`、`留存`、`分群`
- 用户明确要求 `竞品分析`、`对标`、`差异化`
- 用户明确要求 `问卷设计`、`survey`、`NPS`、`满意度调研`

这些场景必须切到对应子技能，不要继续停留在本技能里输出通用建议。
评审类任务切到 `zz-pm-review-board`，埋点类任务切到 `zz-pm-tracking-spec`，实验类任务切到 `zz-pm-experiment`，复盘类任务切到 `zz-pm-postmortem`，优先级类任务切到 `zz-pm-prioritization`，路线图类任务切到 `zz-pm-roadmap`，分析类任务切到 `zz-pm-analytics`，竞品类任务切到 `zz-pm-competitor`，问卷类任务切到 `zz-pm-survey`，原型类任务切到 `zz-pm-prototype-workflow`，文档交付类任务切到 `zz-pm-prd`。

未命中子技能时，本技能负责输出：

- 当前任务属于哪个阶段
- 下一步该看哪份文档
- 当前应先补什么输入
- 当前为什么不应直接进入 PRD 或原型

## 与 zz-pm-prd 的边界

`zz-pm-workflow` 负责：

- 流程判断
- 输入归拢
- 文档优先级判断
- 任务分流

`zz-pm-prd` 负责：

- 初评版 PRD
- 正式交付版 PRD
- 模板套用
- PRD 自检和交付

规则：

- 不在本技能里直接写完整 PRD
- 一旦确认任务进入 PRD 阶段，立即切换到 `zz-pm-prd`

## 输出要求

当本技能被触发时，优先输出：

1. 当前任务类型
2. 当前所处阶段
3. 应优先读取的文档 / 技能
4. 下一步动作

如果信息不足，输出：

- 已知信息
- 待确认项
- 建议先进入的流程节点

## 标准输出格式

默认使用以下结构输出，除非用户明确要求别的格式：

```markdown
## 当前判断
- 任务类型：
- 当前阶段：
- 是否需要切子技能：

## 应先读取
- 文档 / 规范 / 技能：

## 下一步动作
- 先做什么：
- 暂时不要做什么：

## 待确认项
- ...
```

要求：

- 输出先判断，再行动
- 不要一上来直接写长文档
- 不要在任务仍模糊时假装进入正式 PRD
- 若可以继续推进，只给单一推荐路径，不同时抛多个等价方案

## 常见误用

以下是本技能应主动避免的误用：

- 用户还在收口阶段，就直接进入完整 PRD
- 用户只想判断流程，却直接输出大段方案正文
- 原型、设计规范、PRD 三类问题混在一起时，没有先分层
- 明明已经命中 `zz-pm-prd`，却仍在本技能中继续写正式文档
