# 版本与路线图规划

本技能用于版本规划和里程碑编排，不负责单个需求 PRD 或具体用户调研。

适用输入：

- 季度目标 / OKR
- 候选能力项或 Epic
- 团队产能
- 依赖方与硬性窗口

不适用输入：

- 只想做需求优先级排序
- 只想写单个功能文档
- 只需要日常 sprint 任务分配

## 工作目标

输出一张可执行的路线图，而不是泛泛的愿景清单。

默认产出应包含：

1. 目标与范围
2. Epic / 能力拆分
3. 里程碑
4. 依赖与关键路径
5. 风险与缓冲方案

## 默认触发场景

- “帮我做 roadmap”
- “做版本规划”
- “排季度计划”
- “梳理里程碑和依赖”
- “这个项目怎么排节奏”

## 边界

- 需求排序 -> `zz-pm-prioritization`
- 正式 PRD -> `zz-pm-prd`
- 复盘 -> `zz-pm-postmortem`

优先使用本技能的判断标准：

- 输入已经基本知道“要做哪些项”
- 目标是“怎么排节奏、里程碑、依赖和缓冲”
- 输出重点是“版本编排与交付顺序”

不应停留在本技能的场景：

- 用户其实在问“哪些该做、哪些不该做” -> `zz-pm-prioritization`
- 用户只是要单个功能的详细方案 -> `zz-pm-prd`

## 执行流程

### 1. 对齐目标

先明确：

1. 目标是什么
2. 成功标准是什么
3. 哪些是 must-have
4. 最大约束是什么

### 2. 拆能力单元

将目标拆成可交付的 Epic / Feature，粒度以 1 到 3 周为宜。

### 3. 编排里程碑

至少明确：

- 里程碑名称
- 时间窗口
- 交付物
- 成功标准
- 依赖项

### 4. 做产能与依赖检查

检查：

- 是否超出团队容量
- 是否有跨团队阻塞
- 哪条是关键路径
- 是否预留了缓冲

### 5. 输出版本建议

至少区分：

- Now
- Next
- Later

## 标准输出格式

```markdown
## 目标与约束
- 季度目标：
- 成功标准：
- 约束：

## 能力拆分
| # | Epic / 能力项 | 目标 | 预估周期 | 优先级 |
|---|---|---|---|---|

## 里程碑
| 里程碑 | 时间 | 交付物 | 依赖 | 风险 |
|---|---|---|---|---|

## 路线图建议
### Now
- ...

### Next
- ...

### Later
- ...

## 风险与缓冲
- ...
```

## 常见误用

- 没有目标就直接排时间
- 不算产能就塞满路线图
- 把优先级讨论和路线图编排混在一起
- 只给时间表，不解释依赖和关键路径
