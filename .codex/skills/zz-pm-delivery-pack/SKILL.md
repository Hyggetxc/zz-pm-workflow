# 研发交付包

**EN Summary:** Builds a dev-ready delivery pack from a confirmed PRD, including proposal, spec, design, and tasks.

## 定位

本技能用于在 `PRD 已确认`、`评审结论已收敛` 之后，整理研发开工所需的核心交付物。

V1 默认只覆盖 4 份文档：

1. `proposal.md`
2. `spec.md`
3. `design.md`
4. `tasks.md`

本技能不替代 `zz-pm-prd`，也不负责原型生成、评审执行或埋点设计。

## 适用输入

适用输入：

- 已确认的 `PRD`
- 评审结论、评审纪要、待补充结论
- 原型、截图、页面方案、流程图
- `DESIGN.md`、设计规范、原型规范、项目 `SOP`
- 需求变更说明与 PRD 更新内容

不适用输入：

- 只有一句模糊需求，尚未形成 `PRD`
- `PRD` 仍处于首次收口阶段，尚未评审
- 用户要从 0 开始写 `PRD`
- 用户要做原型、复刻页面、输出 `.pen`

## 默认产出

默认产出为一组面向研发交付的 Markdown 文档：

### 1. `proposal.md`

回答为什么做、做什么、不做什么、页面范围和关键边界。

### 2. `spec.md`

回答对象、字段、状态、规则、校验、异常和发布生效逻辑。

### 3. `design.md`

回答页面结构、交互、状态、弹窗、边界和原型映射。

### 4. `tasks.md`

回答产品、前端、后端、测试分别要做什么，以及发布前的对齐检查。

## 边界

`zz-pm-workflow` 负责：

- 判断当前任务是不是研发交付包场景
- 判断应优先读取哪些项目文档
- 将任务路由到本技能

`zz-pm-prd` 负责：

- 生成 `PRD`
- 从初评版推进到正式版 `PRD`

`zz-pm-review-board` 负责：

- 对 `PRD`、原型或方案做评审前检查
- 输出阻断项、重要项、建议项和复评清单

本技能负责：

- 基于已确认输入生成交付四件套
- 需求变更后按影响范围更新四件套
- 保持四份文档之间的术语、范围和状态定义一致

规则：

- 若用户要“写 PRD / 生成正式版 PRD”，切换到 `zz-pm-prd`
- 若用户要“评审 / review / 查漏补缺”，切换到 `zz-pm-review-board`
- 若用户要“proposal / spec / design / tasks / 研发交付包”，使用本技能

## 默认触发场景

以下任务优先使用本技能：

- “PRD 评审完了，帮我出 proposal、spec、design、tasks”
- “把这份已确认 PRD 整理成研发交付包”
- “需求变更了，PRD 已经更新，帮我同步 proposal/spec/design/tasks”
- “不要只给 PRD，要给前后端能直接开工的交付物”
- “按模板把研发交付包补齐”

## 执行流程

### 1. 先判断是首次交付还是变更更新

优先判断：

1. 当前是首次把 `PRD` 转成交付包
2. 还是已有交付包，需要按变更做增量更新

若是变更更新，先提取：

- 变更点
- 影响模块
- 受影响页面 / 状态 / 规则
- 必须同步更新的文档范围

### 2. 核对前置条件

至少需要满足以下条件中的前两项：

1. 有已确认的 `PRD`
2. 有评审结论或明确确认口径
3. 有原型、页面方案或截图
4. 有项目规则、设计规范或 `DESIGN.md`

若缺少会直接影响写作的关键信息，只提最少的问题。

默认只把以下信息视为阻断项：

- 当前改版目标不清楚
- 改版范围不清楚
- 关键页面 / 模块范围不清楚
- 关键状态或规则不清楚

非阻断缺口可先生成文档，并标记 `待确认`。

### 3. 建立交付一致性骨架

先抽出四份文档都要复用的事实层：

- 变更名称 / 版本号
- 目标与背景
- 本期包含 / 不包含
- 页面 / 模块范围
- 统一状态模型
- 关键规则与边界
- 研发分工与发布前检查点

这一步的目标是保证：

- `proposal` 不脱离事实
- `spec` 不发散到无关背景
- `design` 不遗漏交互与状态
- `tasks` 不脱离前三份文档

### 4. 按固定顺序生成文档

默认顺序：

1. `proposal.md`
2. `spec.md`
3. `design.md`
4. `tasks.md`

原因：

- `proposal` 先锁目标、范围和边界
- `spec` 再锁规则、状态和字段
- `design` 再锁页面与交互
- `tasks` 最后把前三份文档翻译成执行动作

### 5. 变更更新时只改受影响文档

若是需求变更，不要默认四份文档全部重写。

优先按下列规则判断：

- 目标、范围变了 -> 更新 `proposal.md`
- 字段、规则、状态变了 -> 更新 `spec.md`
- 页面、交互、弹窗、状态表现变了 -> 更新 `design.md`
- 实施动作、角色分工、检查项变了 -> 更新 `tasks.md`

若多项同时变化，再更新多份文档。

### 6. 输出前自检

至少检查以下内容：

1. 四份文档中的模块名、页面名、状态名是否一致
2. `proposal` 是否只讲目标、范围和边界，没有写进太多字段细节
3. `spec` 是否把字段、状态、规则和异常写清楚
4. `design` 是否覆盖主流程、状态、弹窗和边界
5. `tasks` 是否可执行，而不是重复文档标题
6. 变更更新时，是否只改了受影响部分，没有无意义重写

## 输出要求

优先输出：

1. 当前模式：首次交付 / 变更更新
2. 已读取输入
3. 交付范围
4. 缺失阻断项
5. 生成结果或拟更新文档清单

要求：

- 术语统一
- 范围边界明确
- 待确认项具体可追踪
- 不把 `PRD` 原文机械复制到四份文档
- 不把交互细节写进 `proposal`
- 不把抽象背景大段写进 `spec`

## 标准输出格式

```markdown
## 交付模式
- 类型：首次交付 / 变更更新
- 交付对象：研发四件套

## 已读取输入
- PRD：
- 评审结论：
- 原型 / 截图：
- 规则 / 规范：

## 当前缺口
- 阻断项：
- 非阻断待确认：

## 计划生成 / 更新
- proposal.md：
- spec.md：
- design.md：
- tasks.md：
```
