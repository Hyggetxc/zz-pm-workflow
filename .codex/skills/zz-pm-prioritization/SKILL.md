# 优先级排序

本技能用于需求与项目优先级决策，不负责正式 PRD、详细路线图或日常排期。

适用输入：

- 候选需求列表
- 当前业务目标
- 团队资源约束
- deadline、依赖、风险

不适用输入：

- 只需要单个需求文档
- 只想排具体周计划
- 只有一个需求，没有比较对象

## 工作目标

输出可在评审会上讲清楚的优先级结果。

默认产出应包含：

1. 候选项清单
2. 评分模型选择
3. 排序结果
4. 取舍理由
5. 建议进入的版本层级

## 默认触发场景

- “帮我排优先级”
- “这些需求先做哪个”
- “用 RICE / ICE / Kano 看一下”
- “资源有限该砍哪些”
- “哪些进本版本”

## 边界

- 写文档 -> `zz-pm-prd`
- 做版本与里程碑编排 -> `zz-pm-roadmap`
- 纯调研问卷 -> `zz-pm-survey`

优先使用本技能的判断标准：

- 输入是“多个候选项之间如何取舍”
- 目标是“先做谁、后做谁、哪些先不做”
- 输出重点是“排序与取舍理由”

不应停留在本技能的场景：

- 已经确定做哪些项，只需要排版本节奏 -> `zz-pm-roadmap`
- 只有 1 个方案，没有比较对象 -> 先回到 `zz-pm-workflow`

## 执行流程

### 1. 整理候选项

至少明确：

1. 名称
2. 一句话目标
3. 受影响用户
4. 预估成本
5. 当前状态

### 2. 选择评分模型

默认建议：

- 数据相对充分：`RICE`
- 早期或信息不全：`ICE`
- 需要看满意度结构：`Kano`
- 明确财务取舍：成本收益

### 3. 打分与排序

要求：

- 每个分数都要有一句理由
- 标记数据支撑与假设
- 分歧大的候选项单独说明

### 4. 做权重校准

优先考虑：

- 当前业务目标
- 合规或硬性 deadline
- 依赖和风险
- 团队容量

### 5. 输出取舍建议

至少输出：

- 现在做
- 下个版本考虑
- 暂缓 / 不做

## 标准输出格式

```markdown
## 输入约束
- 业务目标：
- 资源约束：
- 硬性条件：

## 模型选择
- 主模型：
- 选择原因：

## 评分结果
| # | 候选项 | 评分 | 排名 | 主要理由 |
|---|---|---|---|---|

## 取舍建议
### 现在做
- ...

### 下个版本考虑
- ...

### 暂缓 / 不做
- ...

## 风险与待确认项
- ...
```

## 常见误用

- 只给排名，不解释原因
- 有硬性约束却仍按纯分数排序
- 把路线图编排混进优先级技能
- 把“现在做什么”与“什么时候交付”混成同一问题
