# 多角色评审

**EN Summary:** Multi-role review for PRDs, prototypes, and solution drafts before formal review.

## 定位

本技能用于评审前检查，不负责重写整份 PRD，也不替代专项设计走查或代码审查。

## 适用输入

适用输入：

- `PRD`
- 原型链接、原型截图、页面方案
- 功能说明、需求稿、评审稿

不适用输入：

- 只有一句模糊需求，尚未形成可评审材料
- 纯代码实现问题
- 纯视觉设计稿润色

## 默认产出

默认应先发现会阻塞上线或阻塞评审的问题，再给出清晰的补充项和复评清单。

默认覆盖 6 个角色视角：

1. 产品经理
2. 研发工程师
3. 测试工程师
4. UI/UX 设计师
5. 运营
6. 法务 / 合规

## 边界

`zz-pm-workflow` 负责：

- 判断当前是不是评审类任务
- 判断应先读哪些项目文档
- 将任务路由到本技能

本技能负责：

- 多角色评审
- 风险分级
- 输出结论、问题清单、复评清单

`zz-pm-prd` 负责：

- 生成或重写正式 PRD

规则：

- 若用户要“写 PRD / 重写文档 / 出正式版”，切换到 `zz-pm-prd`
- 若用户要“评审 / review / 查漏补缺 / 过会前检查”，使用本技能

## 默认触发场景

以下任务优先使用本技能：

- “帮我评审一下这个 PRD”
- “这个方案能不能过评审”
- “帮我查漏补缺”
- “从研发/测试/设计视角看一下”
- “这个原型有没有风险”
- “评审前先帮我过一遍”

## 执行流程

### 1. 先判断材料是否足够评审

先确认至少具备以下两类信息中的一类：

- 页面 / 功能范围
- 主要流程 / 交互 / 规则

若材料不足，先输出：

- 已知信息
- 缺失信息
- 暂不建议评审的原因

不要在信息严重不足时假装完成评审。

### 2. 提取评审上下文

优先提取：

1. 文档类型
2. 目标用户
3. 核心流程
4. 范围边界
5. 关键依赖
6. 上线或评审阶段

### 3. 分角色审查

#### 产品经理视角

检查：

- 目标是否清晰、可判断
- 用户场景是否闭环
- 范围是否清楚
- 待确认项是否暴露充分
- 是否存在范围蔓延

#### 研发工程师视角

检查：

- 数据结构、字段、状态是否足够清楚
- 外部依赖、接口、权限是否明确
- 异常和并发是否覆盖
- 是否存在明显不可实施点
- 是否缺少开发前置条件

#### 测试工程师视角

检查：

- 验收标准是否可执行
- 主流程、异常流程、边界值是否覆盖
- 状态流转是否闭环
- 是否能直接转成测试用例

#### UI/UX 设计师视角

检查：

- 是否符合 `DESIGN.md` 与设计规范
- 页面状态是否完整
- 操作反馈是否明确
- 信息层级是否合理
- 是否缺少空态、加载、错误、成功、禁用状态

#### 运营视角

检查：

- 是否定义核心埋点
- 是否有灰度、上线、回滚考虑
- 是否有运营配置需求
- 是否能追踪关键漏斗

#### 法务 / 合规视角

检查：

- 是否涉及个人信息、权限、告知与同意
- 是否存在敏感数据与留存风险
- 是否涉及内容审核、版权或第三方合规问题
- 是否需要补充协议、声明、审计要求

### 4. 合并与分级

相同问题合并，不同角色共同提出时合并标注角色来源。

风险分级：

- `阻断项`：不解决不应进入开发或上线
- `重要项`：应在评审通过前补齐
- `建议项`：不阻塞当前阶段，但建议补充

### 5. 输出结论

结论只允许三种：

- `通过`
- `有条件通过`
- `不通过`

若存在任一 `阻断项`，默认不通过或有条件通过，不要轻易给“通过”。

## 输出要求

优先输出：

1. 总体结论
2. 阻断项
3. 重要项
4. 建议项
5. 各角色评审意见
6. 复评清单

要求：

- 问题必须具体，避免空泛措辞
- 尽量标出对应章节、页面、流程或状态
- 修改建议必须可执行
- 不重复罗列等价问题

## 标准输出格式

```markdown
## 评审结论
- 总体判定：
- 一句话原因：

## 通过条件
- ...

## 不可上线 / 不可过评审条件
- ...

## 阻断项
| # | 问题 | 角色 | 所在位置 | 修改建议 |
|---|---|---|---|---|

## 重要项
| # | 问题 | 角色 | 所在位置 | 修改建议 |
|---|---|---|---|---|

## 建议项
| # | 问题 | 角色 | 所在位置 | 修改建议 |
|---|---|---|---|---|

## 各角色意见
### 产品
- ...

### 研发
- ...

### 测试
- ...

### 设计
- ...

### 运营
- ...

### 合规
- ...

## 复评清单
| # | 修改项 | 对应问题 | 复评标准 | 状态 |
|---|---|---|---|---|
```

## 常见误用

避免以下情况：

- 用户还在需求收口阶段，就直接进入完整评审
- 用户要的是重写 PRD，却只输出 review 结论
- 只有截图或只言片语，却给出过度确定的结论
- 只列风险，不给修改建议
- 已存在阻断项，却仍然给“可直接上线”
