# 上线与项目复盘

**EN Summary:** Creates structured postmortems for launches, projects, incidents, and follow-up actions.

## 定位

本技能用于复盘，不负责写 PRD、排优先级或生成周报。

## 适用输入

适用输入：

- 版本目标与实际结果
- 上线数据、问题清单、用户反馈
- 项目时间线、关键决策、事故记录

不适用输入：

- 只有需求描述，没有结果数据或过程记录
- 纯状态汇报
- 纯归责讨论，不需要结构化改进动作

## 默认产出

默认应将一轮项目、版本或事故复盘成可执行的结论与行动项。

默认产出应包含：

1. 复盘范围与背景
2. 目标与结果对照
3. 偏差归因
4. 经验沉淀
5. 行动项与 owner

## 边界

`zz-pm-workflow` 负责：

- 判断当前是不是复盘类任务
- 判断应先读取哪些项目文档
- 将任务路由到本技能

本技能负责：

- 目标结果对照
- 偏差归因
- 经验沉淀
- 行动项输出

`zz-pm-prd` 负责：

- 需求与方案文档产出

规则：

- 若用户要写需求文档本身，切到 `zz-pm-prd`
- 若用户要做评审前检查，切到 `zz-pm-review-board`
- 若用户要做复盘、RCA、行动项沉淀，使用本技能
- 若用户只是要解释数据现象或做漏斗/留存分析，切到 `zz-pm-analytics`

优先使用本技能的判断标准：

- 输入核心是“已经发生了一轮项目、版本、上线或事故”
- 目标是“对照目标、归因、沉淀经验、生成行动项”
- 输出重点是“责任清晰的后续改进”

## 默认触发场景

以下任务优先使用本技能：

- “写复盘”
- “上线复盘 / 项目复盘 / 版本复盘”
- “postmortem”
- “帮我整理复盘报告”
- “这次上线问题帮我做 RCA”
- “把目标、结果和问题整理成行动项”

## 执行流程

### 1. 先确认复盘范围

优先确认：

1. 复盘对象
2. 时间范围
3. 原始目标
4. 实际结果
5. 关键问题或异常

若缺少目标或结果数据，先输出：

- 已知信息
- 缺失输入
- 暂不建议直接定结论的原因

### 2. 做目标与结果对照

至少输出：

- 目标项
- 目标值
- 实际值
- 偏差
- 达成状态

状态建议统一为：

- `达成`
- `部分达成`
- `未达成`
- `超额达成`

### 3. 做偏差归因

针对每个未达成或部分达成项，至少回答：

1. 发生了什么
2. 为什么发生
3. 根因属于哪一类
4. 哪些是可控因素
5. 责任角色是谁

归因时避免：

- 只写表面原因
- 只写“沟通问题”这类空话
- 不区分可控与不可控

### 4. 提炼经验沉淀

默认拆成三类：

- 做对了什么
- 做错了什么
- 可复用的方法论

### 5. 生成行动项

每条行动项必须包含：

1. 具体动作
2. 关联问题
3. Owner
4. 截止日期
5. 完成标准

没有 owner 或截止时间的行动项，默认不合格。

### 6. 做质量检查

输出前至少检查：

- 是否有量化目标和结果
- 偏差是否有数据支撑
- 根因是否足够具体
- 行动项是否可执行
- 是否存在空话和无法验证的结论

## 标准输出格式

```markdown
## 复盘范围
- 对象：
- 时间范围：
- 参与角色：

## 目标达成情况
| # | 目标项 | 目标值 | 实际值 | 偏差 | 状态 |
|---|---|---|---|---|---|

## 偏差归因
| # | 偏差项 | 根因 | 根因类型 | 可控性 | 责任方 |
|---|---|---|---|---|---|

## 经验沉淀
### 做对了什么
- ...

### 做错了什么
- ...

### 方法论沉淀
- ...

## 行动项
| # | 行动项 | 关联问题 | Owner | 截止日期 | 完成标准 |
|---|---|---|---|---|---|

## 待确认项
- ...
```

## 输出要求

要求：

- 先事实，后判断
- 结论必须能追溯到目标、结果或过程记录
- 行动项必须能被跟踪
- 若数据不足，明确标记假设或待确认项

## 常见误用

避免以下情况：

- 把复盘写成流水账
- 只有总结，没有归因
- 只有归因，没有行动项
- 行动项没有 owner、日期或完成标准
- 用户其实要写 PRD 或评审文档，却只输出复盘报告
- 用户只是想分析数据波动，却直接上升为完整复盘
