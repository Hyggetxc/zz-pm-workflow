# 产品文档生成

用于通用产品文档生成场景。保留“按业务事实落地 PRD”的能力，同时支持一个更实用的两阶段工作流：

1. 先形成可评审的原型或页面方案
2. 基于原型或页面方案输出 `初评版 PRD`
3. 评审收敛后升级为 `正式交付版 PRD`

## 默认执行模式

当用户只说“写文档 / 生成 PRD / 按目录出文档”时，默认执行：

1. 读取目录中的样本、截图、原型、规则。
2. 建立风格画像，包括结构、术语、序号、颗粒度。
3. 若用户未明确文档阶段，默认先输出 `初评版 PRD`。
4. 默认优先输出本地 `docx`；若环境不支持，再降级为 `Markdown`。
5. 输出前必须执行自检，不合格页面重写。

## 文档分阶段规则

### 阶段 A：初评版 PRD

默认使用场景：

- 原型已完成或页面方案已具备
- 正在进行首次产品评审
- 需要围绕业务流程、页面修改点和待确认项收敛方案

输出目标：

- 为首次评审服务
- 强调核心业务流程、涉及页面修改、每页待确认项、关键逻辑交互

规则：

1. 优先引用当前原型、页面方案或相关截图。
2. 允许保留待确认项，不强求一次写死全部细节。
3. 优先写评审真正要讨论的内容，不堆正式交付信息。
4. 语言以清晰、可评审、可落地修改为准。

### 阶段 B：正式交付版 PRD

使用场景：

- 原型已确认或已较完整
- 需要给设计、研发或交付相关角色输出更正式文档

输出目标：

- 形成正式 PRD 或功能说明文档

规则：

1. 优先附页面截图或原型截图。
2. 更严格执行 `4.1 -> 4.2 -> 5` 的结构约束。
3. 命中流程图条件时必须补流程图。
4. 文风更正式，适合最终交付。

## 交付规则

1. 默认输出本地 `docx`。
2. 仅在无法生成 `docx` 时回退 `Markdown`。
3. 若用户明确要求在线文档平台，再切换到对应平台流程。
4. 未明确要求时，不主动绑定特定在线文档平台。

## 输入约定

优先读取，存在即用：

- `examples/`：历史高质量文档
- `screenshots/`：页面截图
- `prototype/`：原型源码或页面入口
- `rules.md`：硬规则，优先级最高
- `glossary.md`：术语表
- `scope.md`：页面范围

详细参考：`references/input-contract.md`

## 内置模板

本技能内置两份模板：

- `references/初评版-PRD-模板.md`
  - 用于首次评审阶段
  - 聚焦核心业务流程、涉及页面修改、每页待确认项、关键逻辑与交互

- `references/正式版-PRD-模板.md`
  - 用于正式交付阶段
  - 结构更完整，适合正式评审与对外协作

使用规则：

1. 若用户未明确阶段，默认使用 `初评版-PRD-模板.md`
2. 若原型与评审结论已较稳定，再切到 `正式版-PRD-模板.md`

## 输出结构

### A. 初评版 PRD

1. 文档信息
2. 背景与目标
3. 核心业务流程
4. 涉及页面与修改范围
5. 页面详细说明
6. 关键逻辑与交互
7. 每页待确认项
8. 初评结论与后续动作

### B. 正式交付版 PRD

1. 文档信息
2. 项目背景与目标
3. 用户场景与用户故事
4. 信息架构与功能清单
5. 功能详细说明
6. 非功能性需求
7. 复杂字段深写规则
8. 待补充信息

## 自检要求

输出前至少检查以下内容：

1. 标题层级是否一致。
2. `4.1` 和 `5` 章是否对齐。
3. 功能清单是否误写成页面内原子操作。
4. 字段是否出现“取自字典”“用于标识”之类空话。
5. 截图缺失时，是否错误使用正式定稿语气。
6. 待确认项和待补充信息是否明确、可执行。
