# 产品文档生成

用于当前项目补充输出正式版 PRD。OpenSpec 仍是产品变更主流程；本技能负责生成对外同步、评审交付或归档用的正式需求说明。

## 默认行为

1. 默认输出 `正式版 PRD`。
2. 默认只输出本地 Markdown（`.md`），不生成 `docx`，不主动询问飞书。
3. 默认使用 `references/正式版-PRD-模板.md`。
4. 若存在 OpenSpec 变更目录，先读取 proposal / design / tasks / specs，再写 PRD。
5. 基于已有 PRD 增量修改时，先确认改动点，再局部同步，不默认整篇重写。

## 输入优先级

存在即读取：

- `rules.md`：硬规则，优先级最高
- `scope.md`：本次覆盖范围
- `glossary.md`：术语表
- `screenshots/`：页面或模块截图
- `prototype/`：HTML / Pencil / 截图等原型来源
- `examples/`：历史高质量文档

PRD 中必须写清原型来源是 HTML、Pencil（`.pen`）还是截图；节点 ID 只能作为补充引用，不能替代业务路径。

## 输出结构

正式版 PRD 默认结构：

1. 文档信息
2. 项目背景与目标
3. 用户场景与用户故事
4. 信息架构与功能清单
5. 功能详细说明
6. 非功能性需求
7. 待补充信息

可选附加交付：

- `AI 实现依据`：仅当用户明确要求“AI 实现依据”“给 AI coding 用”“拆实现影响范围”时另行输出独立 Markdown。不因 PRD 面向研发评审而默认生成。

## 第 4 章规则

### 4.1 信息架构

使用代码块树形结构。第 `5` 章标题层级必须与 `4.1` 对齐。

### 4.2 功能清单

固定表头：

- `平台 | 一级功能 | 二级功能 | 优先级 | 说明`

规则：

1. 只写模块级能力，不写页面内原子操作。
2. 二级功能优先对应页面、页签或独立业务模块。
3. 上架、下架、删除、导出、下载、设为默认、筛选、重置等动作写入第 `5` 章，不在 `4.2` 单独占行。
4. 说明列写业务能力边界和核心约束，不写实现细节。

## 第 5 章规则

第 `5` 章是唯一完整需求描述区。第 `5` 章里的每个子模块都是一个独立需求单元；复杂大需求可以在该子模块下继续拆分。

必须遵守：

1. 按需求单元组织，不按字段、异常、流程图、状态机等说明类型单独成章。
2. 前端展示、交互、字段校验、用户提示归入对应 `5.x` 需求单元。
3. 认证、状态判断、额度限制、流量判断、心跳下线、资源分配、统计口径等后端业务逻辑归入对应 `5.x` 需求单元。
4. 字段规则、异常边界、状态流转、验收口径、流程图只能作为 `5.x` 需求单元内部表达方式。
5. 同一条业务判断只能在一个位置完整描述；若多个需求单元共用同一口径，只在第一个相关需求单元完整描述，其他需求单元引用并说明差异。

每个 `5.x` 需求单元按需包含：

1. 需求标题
2. 原型截图或来源说明
3. 功能详设表或分段说明
4. 前端说明
5. 后端逻辑
6. 异常边界
7. 流程图
8. 验收口径

说明：

- 无前端影响时可省略前端说明；无后端影响时可省略后端逻辑。
- 流程图仅在多分支判断、阻断校验或关键状态流转时补充，且必须放在对应 `5.x` 需求单元内。
- 非功能性要求只放真正跨全局的性能、稳定性、兼容性、安全或交付要求。

## 写法规则

### 功能详设表

默认四列：

- `功能模块 | 元素 | 实现归属 | 逻辑说明`

写法：

1. 先模块后字段，不要一上来逐字段平铺。
2. 元素可按从上到下、从左到右描述。
3. 字段句式使用：`字段名，字段类型，字段要求`。
4. `实现归属` 必填，取值只能是 `前端`、`后端`、`前后端`、`非代码项`。
5. 前端展示、样式、交互、提示、空态、禁用态写 `前端`。
6. 认证、计算、服务端校验、额度、权限、状态变更、数据统计、历史数据处理写 `后端`。
7. 前端提交或展示依赖后端判断、保存、返回结果的逻辑写 `前后端`。
8. 运营约定、上线依赖、人工补充、不涉及代码实现的说明写 `非代码项`。
9. 通用元素如面包屑、标题、返回按钮，无业务规则时压缩或省略。

### 前端说明

覆盖页面模块、字段、按钮、筛选项、表格列、表单项、提示、弹窗、状态展示，以及点击、提交、重置、跳转、关闭、返回、加载、成功、失败、空态、禁用态。

### 后端逻辑

覆盖触发条件、输入字段、业务校验、计算口径、状态变化、权限判断、额度限制、异步结果、失败结果和是否影响历史数据。

不写数据库表结构、API 路径、接口字段、服务拆分、组件路径、前端状态管理、研发内部重试、队列、缓存等技术实现细节。只有这些信息直接影响用户可见行为或验收边界时，才转译成产品语言。

### 字段规则

状态、时间、数量、比例、开关、单选、阈值等字段，必须在对应 `5.x` 需求单元内写清默认值、校验规则、计算口径、边界处理和影响范围。

原型或代码中有明确枚举值时必须展开；没有明确值时列入待补充信息，不写“取自字典”。

### 异常边界

可用表头：

- `异常场景 | 触发条件 | 用户提示 | 系统处理 | 是否可重试 | 后续处理`

要求：

1. 只写用户可见或业务可验收的异常。
2. 阻断型异常写清是否停留当前页、是否保留已填内容、是否允许重试。
3. 异步失败写清前台状态、后台状态和后续处理路径。

### 流程图

默认生成 Mermaid 源码。节点文案使用中文，按产品视角描述动作、判断和结果。只画关键主链路和关键分支，不把细枝末节塞进图里。

## 场景补充

### 官网 / Landing / SEO / 对比页

重点写清页面目标、目标用户、阅读路径、核心卖点、信任信息、FAQ、CTA、转化路径、SEO 意图和响应式要求。

### 业务后台 / 管理端

重点写清列表、筛选、表格、分页、详情、表单、弹窗、抽屉、字段定义、枚举、校验、权限、数据范围、批量操作、导入导出、审核、日志和状态变更；异常与状态流转写在对应 `5.x` 需求单元内。

## 缺失信息处理

以下信息缺失时，不要假装写清楚：

1. 页面截图
2. 枚举值
3. 状态取值
4. 权限范围
5. 统计口径
6. 历史数据处理方式

能从原型、代码或 OpenSpec 补齐就补齐；不能补齐则列入“待补充信息”。缺失会影响定稿质量时，将对应需求单元标记为“草稿版说明”。

## 质量闸门

输出前必须检查：

1. `4.1`、`4.2` 与第 `5` 章是否对齐。
2. 第 `5` 章是否按需求单元组织。
3. 是否出现独立的复杂字段、异常处理、状态机或流程图章节。
4. 同一条字段、认证、流量、状态或异常规则是否重复完整描述。
5. 后端业务逻辑是否写入对应 `5.x` 需求单元。
6. 功能详设表是否包含 `实现归属`，且取值是否为 `前端`、`后端`、`前后端`、`非代码项`。
7. `4.2` 是否误把原子操作拆成二级功能。
8. 枚举字段是否只写“取自字典”。
9. 是否出现“用于标识 / 用于识别 / 用于展示 / 用于确认”等空话。
10. 截图缺失时是否误用正式定稿语气。
11. 正文是否混入技术实现细节。

## 禁止写法

- `A或B`
- `待研发确认`
- `待定`
- `可选`
- `研发自行决定`
- `取自字典`（未展开枚举时）
- `本次未提供独立截图，页面以源码原型为准`
- 泛化空话，如“系统处理后返回结果”
