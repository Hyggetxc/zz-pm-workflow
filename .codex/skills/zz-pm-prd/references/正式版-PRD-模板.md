# 正式版 PRD 模板

说明：

- 本模板用于正式交付阶段。
- 它强调结构稳定、逻辑可验收、内容可交付。
- 若已有初评版 PRD，正式版应在其确认结论基础上补全细节，而不是重新发散。

## 1. 文档信息

默认使用表格：

| 字段     | 内容             |
| -------- | ---------------- |
| 文档名称 |                  |
| 项目名称 |                  |
| 文档版本 |                  |
| 输出日期 |                  |
| 输出目标 | 在线文档 / 本地 docx |
| 说明     |                  |

## 2. 项目背景与目标

要求：

- 说明项目背景
- 说明业务目标
- 说明本次文档覆盖目的

写法要求：

- 只写业务背景和目标
- 不混入技术实现细节
- 不写泛化空话

## 3. 用户场景与用户故事

要求：

- 描述典型用户场景
- 描述主要用户故事

建议结构：

### 3.1 用户场景

- 场景 1：
- 场景 2：
- 场景 3：

### 3.2 用户故事

- 作为 `<某类用户>`，我希望 `<完成某件事>`，从而 `<获得某种结果>`。

## 4. 信息架构与功能清单

### 4.1 信息架构

必须使用代码块树形结构呈现。

示例：

```text
平台
├── 一级功能 A
│   ├── 二级功能 A1
│   └── 二级功能 A2
├── 一级功能 B
│   ├── 二级功能 B1
│   └── 二级功能 B2
```

要求：

- `5` 章的标题层级必须与这里保持一致
- 不允许后文与信息架构脱节

### 4.2 功能清单

固定表头：

| 平台 | 一级功能 | 二级功能 | 优先级       | 说明 |
| ---- | -------- | -------- | ------------ | ---- |
|      |          |          | P0 / P1 / P2 |      |

硬规则：

1. 只写模块级能力，不写页面内原子操作。
2. 二级功能优先对应页面、页签或独立业务模块。
3. 上架、下架、删除、导出、下载、设为默认、筛选、重置等动作，不在 `4.2` 单独占行。
4. 说明列写“业务能力边界 + 核心约束”，不写实现细节。

## 5. 功能详细说明

要求：

- 本章是正文核心
- 标题层级必须与 `4.1 信息架构` 保持一致
- 不允许把所有页面平铺在同一级

### 5.x 页面输出顺序

每个正式页面按以下顺序输出：

1. 页面标题
2. 原型截图
3. 功能详设表

如果缺少截图：

1. 优先从原型补抓截图
2. 无法补抓时，只能标记为“草稿版页面说明”
3. 不允许用正式定稿语气输出

### 5.x 页面类型默认模板

#### 列表页

默认顺序：

1. 信息汇总
2. 主按钮
3. 筛选区
4. 列表
5. 分页
6. 空状态

#### 表单页

默认顺序：

1. 表单分组
2. 字段定义
3. 提交按钮
4. 取消 / 返回
5. 提交校验
6. 保存成功与失败结果

#### 详情页

默认顺序：

1. 概览信息
2. 页签或二级导航
3. 当前页签主信息
4. 子模块
5. 操作区

#### 占位页

默认不为每个占位导航页单独展开完整章节。

规则：

1. 默认合并成“占位模块汇总说明”
2. 仅在明确要求逐页展开时，才为每个占位页单独成节

### 5.x 功能详设表

每页独立表格，默认三列：

| 功能模块 | 元素 | 逻辑说明 |
| -------- | ---- | -------- |
|          |      |          |

### 5.x 元素列写法

要求：

- 从上到下、从左到右描述
- 但必须先模块后字段
- 不允许一上来按字段平铺

推荐句式：

- `字段名，字段类型，字段要求`

列表、详情、表单模块优先按以下顺序描述：

1. 展示范围
2. 展示顺序
3. 字段说明
4. 字段交互

通用低价值元素规则：

1. 面包屑、页面标题、返回按钮等通用元素，如无业务规则，可压缩描述或省略
2. 优先写筛选区、列表、状态、主按钮、提交按钮、关键弹窗等有业务影响的模块

### 5.x 字段定义规则

1. 原型或代码里有明确枚举值，必须直接展开，不写“取自字典”。
2. 原型或代码里没有明确枚举值，不要假装写清楚，应列入“待补充信息”。
3. 如果字段支持搜索、联想、模糊匹配、精确匹配，必须写清匹配方式。
4. 如果字段影响后续流程、其他页面展示或状态流转，必须补影响范围。

### 5.x 逻辑说明列写法

逻辑说明必须写“可验收业务逻辑”，重点覆盖：

1. 入参校验
2. 命中规则 / 过滤规则
3. 状态流转与触发条件
4. 计算口径与边界处理
5. 异常提示与空态
6. 权限或操作限制，如有

不要写：

1. 技术实现细节
2. 无效目的句，如“用于标识当前页面位置”

### 5.x 技术信息隔离规则

以下内容默认不进入正文功能说明：

1. 路由配置、Mock 数据、组件路径等技术来源
2. URL 参数、前端状态管理实现、接口调用方式

只有在这些内容直接影响用户可见行为时，才允许转译成业务规则。

## 6. 非功能性需求

可按需覆盖：

- 性能要求
- 稳定性要求
- 兼容性要求
- 权限与安全要求
- 交付要求

说明：

- 只写业务侧关心的非功能性要求
- 不展开纯研发实现方案

## 7. 复杂字段深写规则

命中以下任一条件，必须进入深写模式：

1. 状态字段
2. 时间字段
3. 数量、比例、进度字段
4. 开关或单选后会影响后续流程的字段
5. 会影响其他页面展示的字段
6. 有并列规则、阈值、边界值的字段

深写至少覆盖：

1. 默认值
2. 校验规则
3. 计算口径
4. 边界处理

## 8. 交付策略

1. 优先确认输出目标：在线文档或本地 `docx`
2. 选择在线文档时，建议显式提供文档链接或 ID
3. 选择在线文档时，如目标或配置缺失，可先阻断再补齐
4. 选择本地输出时，默认写本地 `docx`
5. 仅当 `docx` 不可用时，降级为本地 `md`

## 9. 待补充信息

建议统一汇总：

1. 页面截图
2. 枚举值
3. 状态取值
4. 权限范围
5. 统计口径
6. 历史数据处理方式
