# 正式版 PRD 模板

说明：

- 本模板代表 当前项目默认的正式版 PRD 输出结构。
- 适用场景：原型与业务逻辑已经较稳定，需要形成可交付、可追踪、可验收的正式文档。
- 当前项目后续 PRD 默认只输出本地 Markdown（`.md`），不输出 `docx`。
- 文件默认命名为 `docs/<版本号或日期> <需求名称>正式版 PRD.md`；若存在 OpenSpec change，可使用 `docs/<change-name> 正式版 PRD.md`。
- 仅当用户明确要求 AI coding 或实现影响拆解时，另行输出 `AI 实现依据` 独立 Markdown 文件，不合并到 PRD 正文。

## 1. 文档信息

默认使用表格：

| 字段     | 内容             |
| -------- | ---------------- |
| 文档名称 |                  |
| 项目名称 |                  |
| 文档版本 |                  |
| 输出日期 |                  |
| 输出目标 | 本地 Markdown（.md） |
| 原型来源 | HTML / Pencil / 截图 |
| 说明     |                  |

## 2. 项目背景与目标

要求：

- 说明项目背景
- 说明业务目标
- 说明本次文档覆盖目的

写法要求：

- 只写业务背景和目标
- 不混入技术实现细节
- 不写泛化空话

## 3. 用户场景与用户故事

要求：

- 描述典型用户场景
- 描述主要用户故事

建议结构：

### 3.1 用户场景

- 场景 1：
- 场景 2：
- 场景 3：

### 3.2 用户故事

- 作为 `<某类用户>`，我希望 `<完成某件事>`，从而 `<获得某种结果>`。

## 4. 信息架构与功能清单

### 4.1 信息架构

必须使用代码块树形结构呈现。

示例：

```text
平台
├── 一级功能 A
│   ├── 二级功能 A1
│   └── 二级功能 A2
├── 一级功能 B
│   ├── 二级功能 B1
│   └── 二级功能 B2
```

要求：

- `5` 章的标题层级必须与这里保持一致
- 不允许后文与信息架构脱节

### 4.2 功能清单

固定表头：

| 平台 | 一级功能 | 二级功能 | 优先级       | 说明 |
| ---- | -------- | -------- | ------------ | ---- |
|      |          |          | P0 / P1 / P2 |      |

硬规则：

1. 只写模块级能力，不写页面内原子操作。
2. 二级功能优先对应页面、页签或独立业务模块。
3. 上架、下架、删除、导出、下载、设为默认、筛选、重置等动作，不在 `4.2` 单独占行。
4. 说明列写“业务能力边界 + 核心约束”，不写实现细节。

## 5. 功能详细说明

要求：

- 本章是正文核心
- 标题层级必须与 `4.1 信息架构` 保持一致
- 不允许把所有页面平铺在同一级
- 第 `5` 章是唯一完整需求描述区
- 第 `5` 章里的每一个子模块都是一个独立需求单元
- 所有前端、后端、交互、异常、状态、流程图和验收口径，必须写入对应需求单元
- 不按字段、异常、流程图、状态机等说明类型单独成章

### 5.x 需求单元输出顺序

每个需求单元按以下顺序输出：

1. 需求标题
2. 原型截图或来源说明（如适用）
3. 功能详设表或分段说明
4. 前端说明（如有前端影响）
5. 后端逻辑（如有后端影响）
6. 异常边界（按需）
7. 流程图（按需，放在本需求单元内）
8. 验收口径

如果缺少截图：

1. 优先从原型补抓截图
2. 无法补抓时，只能标记为“草稿版页面说明”
3. 不允许用正式定稿语气输出

### 5.x 需求类型默认模板

#### 列表页

默认顺序：

1. 信息汇总
2. 主按钮
3. 筛选区
4. 列表
5. 分页
6. 空状态

#### 表单页

默认顺序：

1. 表单分组
2. 字段定义
3. 提交按钮
4. 取消 / 返回
5. 提交校验
6. 保存成功与失败结果

#### 详情页

默认顺序：

1. 概览信息
2. 页签或二级导航
3. 当前页签主信息
4. 子模块
5. 操作区

#### 占位页

默认不为每个占位导航页单独展开完整章节。

规则：

1. 默认合并成“占位模块汇总说明”
2. 仅在明确要求逐页展开时，才为每个占位页单独成节

#### 官网 / Landing / SEO / 对比页

默认重点：

1. 页面目标
2. 目标用户
3. 阅读路径
4. 核心卖点
5. 信任信息 / FAQ
6. CTA 与转化路径
7. SEO 意图或内容覆盖范围（如适用）

#### 业务后台 / 管理端

默认重点：

1. 列表 / 筛选 / 表格 / 分页
2. 详情 / 表单 / 弹窗 / 抽屉
3. 字段定义 / 枚举 / 校验
4. 权限 / 数据范围 / 批量操作
5. 导入 / 导出 / 审核 / 日志（如适用）
6. 异常处理，写在对应需求单元内
7. 状态流转，写在对应需求单元内

### 5.x 功能详设表

每个需求单元独立表格，默认四列：

| 功能模块 | 元素 | 实现归属 | 逻辑说明 |
| -------- | ---- | -------- | -------- |
|          |      |          |          |

`实现归属` 必填，只允许以下取值：

1. `前端`：页面展示、样式、交互、提示、弹窗、空态、禁用态、前端可独立完成的输入限制。
2. `后端`：认证、计算、额度、权限、状态变更、数据统计、历史数据处理、服务端业务校验。
3. `前后端`：前端提交或展示依赖后端判断、保存、返回结果的逻辑。
4. `非代码项`：运营约定、上线依赖、人工补充、不涉及代码实现的说明。

### 5.x 元素列写法

要求：

- 从上到下、从左到右描述
- 但必须先模块后字段
- 不允许一上来按字段平铺

推荐句式：

- `字段名，字段类型，字段要求`

列表 / 详情 / 表单模块优先按以下顺序描述：

1. 展示范围
2. 展示顺序
3. 字段说明
4. 字段交互

通用低价值元素规则：

1. 面包屑、页面标题、返回按钮等通用元素，如无业务规则，可压缩描述或省略
2. 优先写筛选区、列表、状态、主按钮、提交按钮、关键弹窗等有业务影响的模块

### 5.x 字段定义规则

1. 原型或代码里有明确枚举值，必须直接展开，不写“取自字典”。
2. 原型或代码里没有明确枚举值，不要假装写清楚，应列入“待补充信息”。
3. 如果字段支持搜索、联想、模糊匹配、精确匹配，必须写清匹配方式。
4. 如果字段影响后续流程、其他页面展示或状态流转，必须补影响范围。

### 5.x 逻辑说明列写法

逻辑说明必须写“可验收业务逻辑”，重点覆盖：

1. 入参校验
2. 命中规则 / 过滤规则
3. 状态流转与触发条件
4. 计算口径与边界处理
5. 异常提示与空态
6. 权限或操作限制（如有）

不要写：

1. 技术实现细节
2. 无效目的句，如“用于标识当前页面位置”

### 5.x 前端 / 后端 / 交互 / 异常边界

必须写：

1. 前端元素：页面模块、字段、按钮、筛选项、表格列、表单项、提示、弹窗、状态展示。
2. 交互行为：点击、提交、重置、跳转、关闭、返回、加载、成功、失败、空态、禁用态。
3. 后端可感知业务逻辑：校验、分配、计算、状态流转、权限判断、额度限制、异步结果、失败后的用户可见结果。
4. 异常处理：触发条件、页面反馈、系统处理、是否可重试、是否需要人工处理。

默认不写：

1. 数据库表结构。
2. API 路径、接口字段和服务拆分。
3. 前端状态管理、组件路径和代码实现。
4. 研发内部重试、队列、缓存等实现细节。

只有当这些技术信息会直接影响用户可见行为、验收边界或业务规则时，才转译成产品语言写入 PRD。

### 5.x 需求单元完整性与去重

归位规则：

1. 页面展示、交互、字段校验、用户提示归入对应 `5.x` 需求单元。
2. 认证、状态判断、额度限制、流量判断、心跳下线、资源分配、统计口径等后端业务逻辑，归入对应 `5.x` 需求单元。
3. 异常处理、状态流转、字段规则、流程图只能作为 `5.x` 需求单元内部表达方式。
4. 同一条业务判断只能在一个位置完整描述；`4.2` 只写摘要，`5` 章写完整逻辑。

典型修正：

- `is_deleted = 0 才允许认证通过` 应写入 `规则认证与状态治理 / 删除规则认证逻辑`。
- `单规则流量或订单流量耗尽后认证不通过` 应写入 `规则认证与状态治理 / 流量耗尽认证拦截`。
- `删除、异常、失效后心跳踢下线` 应写入 `规则认证与状态治理 / 心跳踢下线逻辑`。
- 如果某个模块很复杂，需要流程图，流程图直接放在该模块内部。

### 5.x 技术信息隔离规则

以下内容默认不进入正文功能说明：

1. `index.html`、路由配置、Mock 数据、组件路径等技术来源
2. URL 参数、前端状态管理实现、接口调用方式

只有在这些内容直接影响用户可见行为时，才允许转译成业务规则。

### 5.x 字段规则写法

命中以下任一条件，必须在对应需求单元内写清字段规则：

1. 状态字段
2. 时间字段
3. 数量、比例、进度字段
4. 开关或单选后会影响后续流程的字段
5. 会影响其他页面展示的字段
6. 有并列规则、阈值、边界值的字段

字段规则至少覆盖：

1. 默认值
2. 校验规则
3. 计算口径
4. 边界处理
5. 并列规则（有则写）
6. 影响范围

位置要求：

1. 优先写入第 `5` 章对应模块。
2. 不得把字段规则抽到独立章节。
3. 若多个需求单元共用同一字段口径，只在第一个相关需求单元完整描述，其他需求单元引用并说明差异。

### 5.x 异常处理写法

后台、订单、资源、分配、支付、权限、导入导出、异步任务等场景，可以在对应需求单元内补异常处理表。

推荐表头：

| 异常场景 | 触发条件 | 用户提示 | 系统处理 | 是否可重试 | 后续处理 |
| -------- | -------- | -------- | -------- | ---------- | -------- |
|          |          |          |          |            |          |

填写要求：

1. 阻断型异常必须写清是否停留当前页、是否保留已填内容、是否允许重试。
2. 异步处理失败必须写清前台状态、后台状态和后续处理路径。
3. 不展开研发内部实现，只写用户可见或业务可验收的异常。
4. 只能写入第 `5` 章对应需求单元。

### 5.x 状态机写法

订单、资源、任务、端口、白名单、套餐、订阅等对象存在多个状态时，可以在对应需求单元内补状态流转说明。

推荐表头：

| 对象 | 当前状态 | 触发动作 | 下一状态 | 允许操作 | 异常状态 |
| ---- | -------- | -------- | -------- | -------- | -------- |
|      |          |          |          |          |          |

填写要求：

1. 状态名称必须来自已确认口径；缺失时列入待补充信息。
2. 每个状态至少写清可见展示和可执行操作。
3. 不确定的状态流转不得写成正式结论。
4. 只能写入第 `5` 章对应需求单元。

### 5.x 流程图写法

以下场景必须补流程图：

1. 多分支判断
2. 阻断校验
3. 关键状态流转
4. 超时、自动结束、撤回、回滚等时序逻辑
5. 多角色协同触发的业务链路

默认输出要求：

1. 默认生成 `Mermaid` 作为流程图中间表示
2. 节点文案使用中文
3. 只画关键主链路和关键分支
4. 流程图必须放在对应需求单元内，标题与需求单元名称保持一致
5. Markdown 交付优先保留 Mermaid 源码，便于后续维护
6. 用户明确要求图片时，再渲染为本地图片并在 `.md` 中引用

## 6. 非功能性需求

可按需覆盖：

- 性能要求
- 稳定性要求
- 兼容性要求
- 权限与安全要求
- 交付要求

说明：

- 只写业务侧关心的全局非功能性要求
- 不写已经能归入某个 `5.x` 需求单元的业务规则
- 不展开纯研发实现方案

## 7. 待补充信息

以下信息缺失时，不要假装写清楚，应统一列入待补充信息：

1. 页面截图
2. 枚举值
3. 状态取值
4. 权限范围
5. 统计口径
6. 历史数据处理方式

建议表头：

| 类别 | 缺失信息 | 影响范围 | 处理建议 |
| ---- | -------- | -------- | -------- |
|      |          |          |          |

## 生成规则（不输出到 PRD 正文）

### 交付策略

默认交付行为：

1. 当前项目默认只输出本地 Markdown（`.md`）
2. 不生成 `docx`
3. 不主动询问飞书文档
4. PRD 交付不切换到 Word / `docx` 输出
5. 仅当用户明确要求 AI coding 或实现影响拆解时，另行输出 `docs/<版本号或日期> <需求名称>AI 实现依据.md`

### 更新策略

默认：

- 局部更新，不整篇重写
- 若对话、原型或已有 PRD 修改过程中出现新的业务规则、页面结构、字段、状态、异常处理或操作路径变更，必须先确认改动点，再局部同步更新 PRD

仅在以下情况整文重写：

1. 用户明确要求“重写整篇 / 全量覆盖”
2. 文档结构严重偏离模板，局部修复成本明显高于重写

### 质量闸门

生成后必须自检。

若出现以下任一情况，必须重写对应章节：

1. `4.2` 中把原子操作拆成独立二级功能
2. `5` 章按字段逐条平铺，没有先按模块组织
3. 出现“用于标识 / 用于识别 / 用于展示 / 用于确认”这类空话
4. 枚举字段只写“取自字典”，没有展开值，也没有列入待补充项
5. 缺少截图却仍以正式定稿语气输出
6. 占位页写得过重，冲淡主链路
7. 正文里混入技术来源和实现细节
8. 命中流程图触发条件却未在对应 `5.x` 需求单元内补流程图
9. 用户明确要求 AI coding 或实现影响拆解，但没有另行输出 `AI 实现依据` 独立文档
10. 后端业务逻辑没有写入对应 `5.x` 需求单元
11. 出现独立的复杂字段、异常处理、状态机或流程图章节
12. 同一条字段、认证、流量、状态或异常规则在多个位置重复完整描述
13. 功能详设表缺少 `实现归属` 列，或 `实现归属` 取值不在 `前端`、`后端`、`前后端`、`非代码项` 范围内

### 禁止写法

以下表达默认禁止直接出现在正式文档中：

- `A或B`
- `待研发确认`
- `待定`
- `可选`
- `研发自行决定`
- `取自字典`
- `本次未提供独立截图，页面以源码原型为准`
- 无口径字段
- 泛化空话，如“系统处理后返回结果”

### 附录素材

仅当用户明确要求附录时，可使用：

- 输入资料目录说明
- 样本文档说明
- 截图来源说明
- 流程图素材说明
