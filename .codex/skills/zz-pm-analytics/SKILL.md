# 数据分析与洞察

**EN Summary:** Turns product data, funnel signals, and retention changes into actionable product insights.

## 定位

本技能用于产品分析，不负责纯 BI 搭建、纯 SQL 编写或埋点设计。

## 适用输入

适用输入：

- 指标报表
- CSV / Excel / SQL 结果
- 看板截图
- 数据问题描述

不适用输入：

- 只想做埋点方案
- 只想做实验设计
- 只有主观判断，没有任何数据或现象

## 默认产出

默认应输出带行动建议的数据分析结论，而不是只描述现象。

默认产出应包含：

1. 分析目标
2. 数据质量判断
3. 关键发现
4. 假设与可信度
5. 建议动作

## 边界

- 埋点方案 -> `zz-pm-tracking-spec`
- 实验设计 -> `zz-pm-experiment`
- 实验复盘 -> `zz-pm-postmortem`

优先使用本技能的判断标准：

- 输入核心是“数据现象、指标问题或分析材料”
- 目标是“解释现象并提出后续动作”
- 输出重点是“发现、假设、可信度和建议”

不应停留在本技能的场景：

- 用户要的是“实验结束后的正式复盘” -> `zz-pm-postmortem`
- 用户要的是“实验如何设计” -> `zz-pm-experiment`
- 用户要的是“如何埋点采集这些数据” -> `zz-pm-tracking-spec`

## 默认触发场景

- “分析一下这组数据”
- “这个指标为什么跌了”
- “帮我做留存 / 漏斗分析”
- “看下用户流失原因”
- “出个分析结论”

## 执行流程

### 1. 明确分析问题

先回答：

1. 要回答什么问题
2. 结果用于什么决策
3. 时间范围和对比基准是什么

### 2. 做数据体检

检查：

- 完整性
- 一致性
- 口径
- 异常值
- 样本量

### 3. 做现象定位

按问题选择：

- 漏斗
- 留存
- 分群
- 时间序列
- 实验结果

### 4. 形成假设

每条结论都要标：

- 支撑证据
- 关系类型
- 可信度
- 建议验证方式

### 5. 输出动作建议

建议应直接指向：

- 产品动作
- 需要补的分析
- 需要做的实验

## 标准输出格式

```markdown
## 分析目标
- 问题：
- 决策场景：
- 时间范围：

## 数据体检
- 完整性：
- 口径：
- 风险：

## 关键发现
| # | 发现 | 支撑证据 | 可信度 |
|---|---|---|---|

## 假设与解释
| # | 假设 | 关系类型 | 验证方式 |
|---|---|---|---|

## 建议动作
- ...
```

## 常见误用

- 直接把相关性当因果
- 不校验口径就下结论
- 只有图表，没有动作建议
- 把“分析现象”直接写成“项目复盘结论”
