# 埋点方案设计

**EN Summary:** Designs event tracking specs with fields, metric definitions, and QA validation checks.

## 定位

本技能用于产出可执行的埋点规范，不负责做 BI 看板、SQL 查询或纯数据分析。

## 适用输入

适用输入：

- `PRD`
- 页面流程、原型、用户旅程
- 核心转化链路
- 现有埋点命名规范或字段字典

不适用输入：

- 只要求分析已有数据结果
- 只要求搭建看板
- 完全没有流程、页面或业务目标描述

## 默认产出

默认应将产品需求转成开发、QA、分析都能直接使用的埋点方案。

默认产出应包含：

1. 埋点范围与目标
2. 核心链路拆解
3. 事件清单
4. 字段定义
5. 指标口径
6. QA 验收清单

## 边界

`zz-pm-workflow` 负责：

- 判断当前是不是埋点类任务
- 判断应先读取哪些项目文档
- 将任务路由到本技能

本技能负责：

- 埋点范围定义
- 事件和字段设计
- 指标口径说明
- QA 验收清单

`zz-pm-prd` 负责：

- 正式 PRD 文档产出

规则：

- 若用户要写需求文档本身，切到 `zz-pm-prd`
- 若用户要补充评审前检查，切到 `zz-pm-review-board`
- 若用户要设计埋点、字段、口径、验收方案，使用本技能

## 默认触发场景

以下任务优先使用本技能：

- “帮我出埋点方案”
- “这个功能要怎么埋点”
- “帮我拆事件和字段”
- “写 tracking spec”
- “定义指标口径和验收清单”
- “这个漏斗怎么设计埋点”

## 执行流程

### 1. 先确认埋点目标

优先确认以下信息：

1. 核心业务目标
2. 要追踪的关键链路
3. 当前阶段是方案设计、开发前、上线前还是补漏
4. 是否已有命名规范、字段字典、数据平台约束

若信息不足，先输出：

- 已知目标
- 缺失信息
- 暂不建议直接出完整方案的原因

### 2. 梳理链路与关键节点

至少拆清楚：

1. 入口
2. 关键操作
3. 状态分支
4. 成功结果
5. 失败结果
6. 退出或流失点

若只有页面没有完整链路，也要先补一个最小链路草图，再往下拆事件。

### 3. 设计事件清单

默认事件类型包括：

- 页面浏览
- 元素曝光
- 元素点击
- 操作开始
- 操作完成
- 操作失败
- 页面停留

规则：

- 同类行为优先通过字段区分，不要为每个按钮单独造新事件
- 成功 / 失败事件尽量成对设计
- 高价值步骤必须可单独追踪

### 4. 设计字段与口径

字段分两层：

- 通用字段
- 业务字段

默认优先定义：

- `event_name`
- `event_time`
- `user_id`
- `device_id`
- `session_id`
- `platform`
- `app_version`

业务字段要求：

- 字段名语义明确
- 类型明确
- 是否必填明确
- 枚举值或示例明确

指标口径要求：

- 指标名称
- 计算方式
- 分子 / 分母定义
- 去重规则
- 时间窗口

### 5. 生成 QA 验收清单

每个关键事件至少给出：

- 触发时机验证
- 去重 / 重复上报验证
- 必填字段验证
- 枚举值验证
- 异常场景验证

### 6. 做一致性检查

输出前至少检查：

- 命名格式是否统一
- 同名字段是否跨事件同义
- 关键漏斗是否存在断点
- 是否只埋成功不埋失败
- 是否缺少关键状态字段

## 标准输出格式

```markdown
## 埋点目标与范围
- 业务目标：
- 覆盖链路：
- 本次不覆盖：

## 链路拆解
| 步骤 | 页面 / 场景 | 用户动作 | 系统结果 | 是否关键节点 |
|---|---|---|---|---|

## 事件清单
| # | 事件名 | 触发时机 | 事件说明 | 优先级 |
|---|---|---|---|---|

## 事件字段定义
### 事件：xxx
| 字段名 | 类型 | 必填 | 说明 | 枚举 / 示例 |
|---|---|---|---|---|

## 指标口径
| 指标 | 计算方式 | 口径说明 | 备注 |
|---|---|---|---|

## QA 验收清单
| 事件 | 验收项 | 校验方式 | 通过标准 |
|---|---|---|---|

## 风险与待确认项
- ...
```

## 输出要求

要求：

- 优先面向开发和 QA 可执行
- 不只列事件名，必须写触发和字段
- 不要把指标口径写成空话
- 若用户未给数据平台，使用平台中立写法
- 若已有命名规范，优先沿用项目规范

## 常见误用

避免以下情况：

- 只有事件列表，没有字段与触发时机
- 只写成功链路，不写失败链路
- 每个 UI 元素都拆成独立事件，导致粒度失控
- 字段名和口径没有统一规则
- 用户其实要写 PRD，却只输出埋点文档
